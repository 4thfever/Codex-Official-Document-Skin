# CODEX 文档模式规格

## 状态

Windows Phase 1 已实现，并通过 renderer 与主题仓库自动化测试。发布前仍需在当前 Codex 任务页完成实机验证。本规格定义非政府化的文档式回复呈现、图片圈阅输入和文档批示回传；不得声称制作、签发或代表政府官方文件。

Windows 是首个目标平台；在可行范围内，renderer 合约应可移植到 macOS 注入器。

## 产品定义

CODEX 文档模式将每条助手回复渲染为正式、克制的 `CODEX` 风格文档，同时保留官方 Codex Desktop 的导航、消息、输入框、项目和任务控制。

功能彼此独立：

1. 回复呈现：将助手回复渲染为文档。
2. 图片圈阅：用户在一张图片上画红圈或红叉，通过 Codex 原生附件路径发送带圈阅的 PNG。
3. 文档批示：用户直接批示已渲染的助手文档，回传包含文档和批示的完整 PNG。

## 非目标

- 不模仿中国政府机构、国徽、印章、文号、密级、官方签发、审批或法律文书。
- 不修改官方 Codex 可执行文件、`app.asar`、WindowsApps、签名、API Key、Base URL、认证或后端请求协议。
- 不修改左侧栏的标签、结构、样式、行为或导航。
- 不把用户消息渲染成文档，也不改写用户输入。
- 首版不支持 PDF、DOCX、OCR、本地 ML、多页批示、协作批示或永久批示存储。
- 不拦截、重写、重放或构造 Codex 上传网络请求；附件后绝不自动发送消息。

## 交互边界

```text
官方 Codex Desktop
    |
    | 本机回环 CDP 注入
    v
CODEX 文档模式
    |- 助手回复文档呈现
    |- 原生输入框旁的图片圈阅
    `- 文档批示覆盖层
```

所有装饰只能注入经验证的 Codex renderer target。装饰层不得拦截原生控件；现有回环 CDP 与 target 身份校验规则仍为强制要求。

## 回复呈现

### 范围

只装饰模型作者消息。优先使用稳定 renderer 标识，例如 `[data-message-author-role="assistant"]` 的祖先节点；不得依据消息文本或位置选择器识别。

每条命中的消息添加幂等文档壳：

```text
美国科代克斯技术服务有限公司
------------------------------------------------
尊敬的董事长：

[原生助手回复内容]

此致

                                      Codex小助手
                                      YYYY年M月D日
```

原生回复 DOM 始终是唯一事实来源。文档壳不得克隆内容、替换消息节点、改变链接、消耗选择行为或改变代码复制控件。

### 视觉与字体

- 题头固定为 `美国科代克斯技术服务有限公司`；`演示稿` 必须保留，以避免声称真实公司签发。
- 使用克制的红色小标宋风格题头及红线；致意固定为 `尊敬的董事长：`，不得推定职务、组织或收件人身份。
- `此致` 左对齐；`Codex小助手` 与本机日期分两行右对齐。这是演示界面落款，不是公司签名、授权、审批或法律签发。
- 纸面为暖白、正文为近黑、边框为低对比中性色；避免沉浸式壁纸和大阴影。
- 文档壳圆角为 `0` 至 `2px`；任务页、侧栏、用户消息、composer 和原生 chrome 保持 Codex 自身外观。
- 流式回复出现消息容器后立即显示文档壳，不闪烁、不重复。

| 内容 | 首选字体 | 回退 |
| --- | --- | --- |
| 演示题头 | 小标宋风格 | 衬线标题字体 |
| 自然语言正文 | 仿宋风格 | `FangSong`、`Noto Serif CJK SC`、serif |
| Markdown H1 | 黑体风格 | `SimHei`、sans-serif |
| Markdown H2 | 楷体风格 | `KaiTi`、serif |
| 行内代码、代码块、命令、JSON、路径 | Cascadia Code / JetBrains Mono | monospace |
| 表格、元数据、紧凑控件 | Codex 原生 UI 字体 | 原有 UI 回退 |

字体必须有授权来源或系统回退。不得将 `pre`、`code`、终端输出、路径、JSON 或 diff 改为仿宋；不得改变文本、标题、列表、链接、复制按钮、表格、选择或键盘导航。原生 Markdown `h1` 可样式化为文档标题，但不得凭空提取或生成标题。

### 主题合约

```json
{
  "schemaVersion": 2,
  "id": "preset-codex-document",
  "name": "CODEX Document",
  "appearance": "light",
  "mode": "codex-document",
  "document": {
    "masthead": "美国科代克斯技术服务有限公司",
    "greeting": "尊敬的董事长：",
    "closing": "此致",
    "signature": "Codex小助手",
    "accent": "#8B1E1E",
    "surface": "#FCFBF7",
    "text": "#24201D",
    "border": "#D8D1C6"
  },
  "annotation": { "enabled": true, "color": "#B40000" }
}
```

`mode: "codex-document"` 禁用本预设原有的全窗口艺术背景。文档模式 CSS 必须限定在助手文档壳中，且不得包含 `aside.app-shell-left-panel` 选择器。

## Phase 2：图片圈阅 MVP

### 工作流与编辑器

用户通过原生 composer 旁的 `圈阅图片` 按钮选择一张 PNG、JPEG/JPG 或 WebP；编辑器显示原图与透明 canvas。用户用红色自由笔画圈、叉、下划线等，可撤销、清空、取消或附加。附加时导出原图与标记合成的新 PNG，经原生附件路径进入 composer，用户自行发送。

工具按钮必须可鼠标和键盘操作，Enter/Space 激活，`aria-label` 为 `圈阅图片`，`pointer-events: auto`；背景装饰保持 `pointer-events: none`。不得移动、替换或遮盖原生附件和发送按钮。

仅支持单张图片：非空、最大 16 MB、任一维度不超过 16384px、总像素不超过 50MP。HEIC、GIF、SVG、PDF、DOCX、多页文件与视频不支持。canvas 应响应式展示、原分辨率导出、处理 DPR 和坐标映射，笔迹不得越界；取消不产生附件，不覆盖源文件。

### 原生附件桥接

导出物为浏览器 `File`。按最小侵入顺序：查找原生文件 input、drop target 或 paste target；优先使用 `DataTransfer` 加原生 `input` / `change` 或 `drop`；确认 composer 可见附件；仅在明确实现和验证后尝试原生剪贴板图片粘贴。失败时不发送、不改网络请求，并保留供用户手动附加。

确认附件可见后，仅追加一次且保留现有草稿：

```text
请结合附件中的红色手绘标记理解我的意见：红圈表示同意，红叉表示不同意。请结合被标记内容和上下文判断；如标记对象或含义不清晰，请明确说明不确定之处。
```

这是可见用户说明，不是隐藏提示词。用户手动删除附件后，MVP 可保留该文本。

## Phase 3：文档批示回传

### 目的与边界

不额外添加手写输入栏，也不改写助手原回复或用户草稿。每条 `.codex-document-response` 就是批示面。完成的文档 hover 时显示画笔光标和 `批示此文` tooltip；用户显式激活后，仅该文档进入批示状态。

批示状态挂载透明、文档局部的 canvas，不安装 document-level pointer handler。未进入批示状态时，链接、复制、选择、滚动和消息操作保持可用。

### 工作流

```text
1. 用户 hover 已渲染助手文档并激活批示。
2. 当前文档出现透明绘制层。
3. 用户手写、画圈、打叉、划线或作其它标记。
4. 用户可撤销、清空、取消或附加。
5. 附加时导出完整渲染文档和批示为一张 PNG。
6. PNG 通过已验证的原生附件路径进入 Codex。
7. composer 追加可见解释说明。
8. 用户查看附件后按 Codex 原生发送。
```

不会自动发送。取消即丢弃笔迹且不产生附件；批示结束后源回复及 live document DOM 保持不变。

### Canvas 与导出

- 导出完整文档，而非仅视口：包含题头、原生回复、结束语、署名、日期和可见批示。
- 不导出侧栏、任务头、composer、其他消息或无关用户内容。
- 使用文档局部、DPR 感知的 canvas，保证笔迹与代码在 PNG 中清晰。
- 只提供红色自由笔，支持撤销上一步、清空、取消和附加；手写文字是笔迹，不是新文本输入控件。
- PNG 只是批示工件，不替代互动原回复，也不改变原生 Markdown/代码渲染。

原生附件可见后仅追加一次：

```text
请结合附件中的文档批示理解我的反馈。若批示包含文字，以文字意见为准；仅有红圈表示同意，仅有红叉表示不同意。请结合被标记内容和上下文判断；如标记对象或含义不清晰，请明确说明不确定之处。
```

这是可见用户说明而非隐藏提示词。手写文字优先；仅在无文字批示时才使用圈叉含义。用户手动删除附件后，MVP 可保留说明文本。

## 实施计划

### Phase 1：助手回复文档壳

- 添加 `preset-codex-document` 主题。
- 扩展 `renderer-inject.js`，识别助手消息，为每条消息幂等挂载题头、致意、结束语、署名与日期。
- 添加限定范围的题头、红线、纸面和字体 CSS。
- pause、remove、reload、restore、reapply 必须清理所有文档节点、类与样式资源。
- 不添加改变左侧栏的选择器。

### Phase 2：图片圈阅 MVP

- 实现 composer 旁图片圈阅按钮、选图、canvas、PNG 导出和清理。
- 探测实时 Codex 附件 DOM 并实现一个经验证的原生附件桥接。
- 附件可见后才追加圈叉解释文本；失败不得清空现有草稿。

### Phase 3：文档批示回传

- 完成的 `.codex-document-response` hover 时显示画笔光标与 `批示此文`，不拦截普通交互。
- 进入每文档独立的透明 canvas 批示状态，并提供撤销、清空、取消、附加工具栏。
- 以 DPR 感知分辨率导出完整文档与批示 PNG，不裁切为视口。
- 复用已验证的原生附件桥接，附件可见后才添加文档批示说明。
- 保留原回复和现有 composer 草稿；不增加独立手写栏、隐藏 wrapper 或自动发送。

## 验收标准

### 回复呈现

- 每条可见助手回复恰有一个演示题头、致意、结束语、署名和日期块；用户消息没有文档样式。
- 左侧栏无视觉或行为变化；代码、终端、diff 和行内代码保持等宽且复制/选择可用。
- 链接、表格、Markdown、按钮和消息操作可交互；流式、路由切换、重载与清理不产生重复或残留。
- 不得出现政府文件、政府行为或官方法律效力声明。

### 图片圈阅

- 按钮可鼠标/键盘使用且不遮挡原生控件；输入限制、笔迹映射、撤销/清空/取消/附加均正确。
- PNG 在原生 composer 可见后才允许发送；说明文本只追加一次且保留草稿。
- 失败不得发送、改网络请求或删除草稿。

### 文档批示

- hover 不阻塞控件；批示模式限于单份文档，支持撤销、清空、取消、附加且无全局 pointer handler。
- 导出 PNG 包含完整文档与红色笔迹，清晰且不含无关应用 chrome。
- 取消或导出后原文档仍互动且不变；原生附件可见后才追加一次说明。
- 文字批示优先；无文字时仅圈表示同意、仅叉表示不同意；歧义标记要求模型说明不确定性。

## 测试策略

- 单元测试 renderer 对助手/用户消息、辅助窗口、流式、路由与清理的幂等性。
- 回归测试禁止文档 CSS 选择侧栏，并验证字体回退与代码等宽字体。
- 测试图片元数据限制、100%/125%/150%/200% DPR 的 canvas 坐标映射与附件桥接 DOM fixture。
- 测试文档局部 canvas 边界、超出视口的完整截取、批示状态清理，以及普通消息控件在非批示状态保持可用。
- 测试导出图包含文档壳、原生回复和笔迹，但不包含侧栏、composer 或其它消息；发布前在真实 Codex 任务页截图验证。

## 待决事项

- 当前 Codex 构建中稳定的助手消息和原生附件选择器。
- 用户手动移除附件时，是否自动移除对应说明文本。
